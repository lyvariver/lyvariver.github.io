我是如何从零开始搭建个人博客的

2025年12月15日 · Lyva 的思考笔记
“最微小的疏忽，往往藏在最基础的原则里。”

为什么我要建一个博客？

我一直想拥有一个真正属于自己的角落——
不依赖微信公众号的算法，不困在小红书的流量池，也不担心某天平台突然改规则。
我想写点东西，就放在那里，安静、持久、可被任何人访问。

于是，我决定用 GitHub Pages 搭建一个纯静态的个人博客。
没有数据库，没有广告，只有 HTML + CSS，和一颗想开始的心。

但没想到，这个看似简单的任务，却让我深刻体会到：
越是基础的事，越容易因一个微小细节而卡住。

第一个“坑”：仓库名字不能随便起！

我以为，只要在 GitHub 上建个仓库，放个 index.html，网站就能自动出来。
于是我创建了仓库：lyvariver/lyvariver。

结果？
访问 https://lyvariver.github.io 时，只看到一行冰冷的提示：

“There isn't a GitHub Pages site here.”

我反复检查文件、刷新页面、甚至怀疑网络……
直到我意识到：GitHub Pages 的个人主页，有且仅有一个命名规则：

✅ 仓库名必须是：你的用户名.github.io

对我而言，就是：lyvariver.github.io

不是 blog，不是 my-site，更不是和用户名同名的 lyvariver。
少一个字母、多一个拼写错误，整个系统就沉默——它不会提醒你，只会让你自己去发现。

这让我明白：技术世界尊重约定，但不提供保姆式引导。
你得主动理解规则，而不是等待提示。

我学会了什么？

经过几天的摸索（其实每天只花一小时），我不仅让博客成功上线，还掌握了一整套“数字安家”的基本技能：

✅ 1. 理解 GitHub Pages 的核心机制

  • 只有 用户名.github.io 仓库能生成 https://用户名.github.io 网址
  • 必须是 Public 仓库，且 index.html 在根目录

✅ 2. 编写结构完整的 HTML 页面

  • 从 <!DOCTYPE html> 到 <footer>，每一块都有意义
  • 设置 <meta charset="UTF-8"> 和 <meta name="viewport"> 确保兼容性

✅ 3. 用 CSS 打造简洁美观的界面

  • 使用 :root 定义主题色变量
  • 通过 @media 实现手机自适应
  • 不依赖任何框架，纯手写样式

✅ 4. 构建多页面导航系统

  • 首页 → 文章页 → 关于页 → 返回首页
  • 所有链接使用相对路径（如 href="about.html"）

✅ 5. 完成部署与调试全流程

  • 在 Settings → Pages 中启用服务
  • 学会等待首次部署（通常 1~5 分钟）
  • 用强制刷新（Ctrl+F5）绕过浏览器缓存

✅ 6. 最重要的：培养“细节敏感度”

  • 文件名大小写敏感（Index.html ≠ index.html）
  • 路径不能错（子文件夹会导致 404）
  • 一个拼写错误，可能浪费一小时排查

这不仅仅是一个博客

现在，当我访问 https://lyvariver.github.io，
看到自己写的文字、设计的布局、搭建的链接，
我感受到一种前所未有的掌控感。

我不再是某个平台的“用户”，
而是自己数字家园的建造者。

这或许就是“数字自主权”的起点：
内容由我写，结构由我定，地址由我拥有。

给同样起步的你

如果你也在尝试搭建博客，请记住：

  • 不要怕犯错：我曾因仓库名错了一个字，折腾半天
  • 每天一小时足够：持续比突击更重要
  • 最小可行产品先行：先让它跑起来，再慢慢美化
  • 把过程写下来：你的踩坑笔记，会成为别人的灯塔

下一步

我会继续在这里记录学习、思考与成长。
也许明天加一篇新文章，下周换个配色，未来甚至绑定自己的域名。

但无论如何，这个小小的网站，已经是我真实存在的证明。

欢迎常来坐坐 🌼
